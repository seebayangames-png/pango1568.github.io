ProfileService DataStore Save System

Complete save/load pipeline using ProfileService — the gold standard for Roblox DataStores. Session-locked profiles, auto-save every 30s, and a strict serialization order that prevents data loss on server shutdown. Saves pets, eggs, enchants, potion timers, rank, currency, and offline state — all in the correct order before profile:Release().

-- Serialization order matters — barn data MUST be written -- before profile:Release() or race conditions drop items. function PlayerDataService:ReleasePlayer(player) local profile = self.Profiles[player.UserId] if not profile then return end self:SyncCurrencyFromAttributes(player) -- 1. live Money/Gems self:_SerializePets(player, data) -- 2. all pets + enchants self:_SerializeEggs(player, data) -- 3. hatch progress self:_SerializePendingEnchants(player, data) -- 4. mid-apply timers self:_SerializeActivePotionEffects(player, data) -- 5. potion timers profile:Release() -- 6. FINAL write — all data guaranteed safe end
  • Potion timers pause offline — stored as timeRemaining, not an expiry timestamp
  • Mid-apply enchants resume on rejoin from exact remaining seconds
  • Race condition fix: PDS waits for ProfileService before any Add() calls

BigNumber Currency System

Handles currency from $1 to $5 Trillion without floating-point loss. Stores values as m×10^e internally, writes "m,e" to attributes so GetAttributeChangedSignal fires on every real change — not broken by abbreviation collisions like "1.25M".

-- "m,e" format: 1,250,000 stored as "1.25,6" -- Fires AttributeChanged on every real addition function CurrencyService:Add(player, currency, amount) local current = self:Get(player, currency) current:Add(amount) self:Set(player, currency, current) end function CurrencyService:Subtract(p, c, amount) if not self:CanAfford(p, c, amount) then return false end -- server-side check — exploit-safe local neg = BigNumber.new(-amount.m, amount.e) self:Get(p, c):Add(neg) return true end
  • Handles values up to 5 Trillion — idle game scale
  • HUD always updates — no silent signal misses from abbreviation collisions

Server-Authoritative Hatch Timer

VirtualElapsed is the single source of truth for egg hatch progress — written only by the server tick loop. Speed potions are applied server-side too. Clients cannot fake progress. Offline hatch credit is applied at 30% speed, capped at 12 hours, on rejoin.

-- Server is SOLE writer of VirtualElapsed. -- Client reads it but never writes — exploit-safe. while egg:GetAttribute("IsPlacedEgg") do task.wait(1) if egg:GetAttribute("HatchPaused") then continue end local mult = owner:GetAttribute("HatchSpeedMultiplier") local ve = egg:GetAttribute("VirtualElapsed") + 1 * mult egg:SetAttribute("VirtualElapsed", ve) if ve >= egg:GetAttribute("HatchDuration") then egg:SetAttribute("ReadyToHatch", true) HatchCompleted:Fire(egg, owner); break end end
  • 5x speed potion = 5 VirtualElapsed seconds per real second
  • Offline progress: 30% credit on rejoin, max 12hr window

Enchant Resume on Rejoin

Enchants take up to 15 minutes to apply. If a player disconnects mid-apply, the remaining time is serialized and the enchant resumes exactly where it left off on the next join.

-- Saves exact TimeRemaining for mid-apply enchants. -- On rejoin, EnchantService.ResumeApplying() picks up -- from the saved seconds — never restarts from zero. if pet:GetAttribute("ApplyingEnchant") == true then local expiry = pet:GetAttribute("ApplyingEnchantExpiry") local remaining = math.max(0, expiry - os.time()) if remaining > 0 then table.insert(saved, { PetName = pet.Name, EnchantName = pet:GetAttribute("ApplyingEnchantName"), TimeRemaining = remaining, StandIndex = standIdx }) end end
  • StandIndex saved — resumes on exact pet, not first same-name match
